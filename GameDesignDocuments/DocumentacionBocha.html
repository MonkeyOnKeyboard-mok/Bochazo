1. Vista General del Sistema

Que es: La escena bocha.tscn representa una bocha de bochas con comportamiento autonomo. Se instancia dentro de una escena padre (la cancha) y maneja todo su ciclo de vida: posicionamiento lateral, apuntado, lanzamiento con gesto del mouse, y rodado hasta detenerse.

Arquitectura: Composicion pura. Cada nodo hijo tiene una responsabilidad unica. No hay un script monolitico. La comunicacion entre componentes se hace por referencias directas (inyectadas en _ready()) y señales.

Patron principal: State Machine con estados como RefCounted (no son nodos). El StateMachine delega _process, _physics_process e _input al estado activo.

Curva/efecto: Se aplica como angular_velocity.y del RigidBody3D. Valor positivo = giro a la derecha, negativo = giro a la izquierda.

La bocha tiene 4 estados que definen su comportamiento en cada momento. Solo un estado esta activo a la vez.

Estado Que hace Freeze? Como se sale
POSITIONING Escucha A/D para mover lateralmente en Z. Escucha click para iniciar lanzamiento. Sì (no cae) Click izquierdo → AIMING
AIMING Escucha gesto del mouse (click→arrastrar→soltar). Calcula potencia y curva. Sì (no cae) Gesto RELEASED con power >= min → THROWING

5. bocha.gd (Nodo Raiz)

Responsabilidad: Orquestador. Cachea referencias a hijos, inyecta referencias cruzadas, y expone señales al exterior.

Variables exportables (se ven en el inspector del nodo Bocha)

VariableTipoDefaultQue controla
lateral_speedfloat3.0Velocidad de movimiento lateral (unidades/segundo)
lateral_boundsVector2(-2.0, 2.0)Limites laterales en Z (min, max)
max_throw_powerfloat15.0Potencia maxima del lanzamiento (fuerza del impulso)
throw_sensitivityfloat1.0Sensibilidad del efecto de curva

Senales (para comunicacion con la escena padre)

SeñalParametrosCuando se emite
bocha_thrown(power: float, direction: Vector3)Cuando se aplica el impulso en ThrowingState
bocha_stopped(ninguno)Cuando la bocha se detiene completamente en RollingState
state_changed(new_state: String)Cada vez que cambia de estado

Flujo de _ready()

func _ready() -> void:
    # 1. Cachear referencias a hijos por nombre
    physics_body = $PhysicsBody
    state_machine = $StateMachine
    position_controller = $PositionController
    throw_controller = $ThrowController
    input_handler = $InputHandler
    visual_effects = $VisualEffects

    # 2. Inyectar referencia al raiz en los controladores
    position_controller.bocha_root = self
    throw_controller.bocha_root = self
    input_handler.bocha_root = self

    # 3. Conectar señal del StateMachine
    state_machine.state_changed.connect(_on_state_changed)

Como modificar

  • Cambiar velocidad lateral: ajustá lateral_speed en el inspector o en el codigo.
  • Cambiar limites laterales: ajustá lateral_bounds. El valor X es el limite minimo (derecha), Y es el maximo (izquierda).
  • Cambiar potencia maxima: ajustá max_throw_power. Afecta directamente la fuerza del impulso.
  • Agregar logica global: este es el lugar correcto para cualquier logica que necesite acceso a todos los componentes.

6. StateMachine.gd (Orquestador de Estados)

Responsabilidad: Gestionar cual estado esta activo y delegar los ciclos de frame (_process, _physics_process, _input) al estado correspondiente.

Como funciona

# Los estados se crean en codigo con .new() (Opcion A)
states[BochaState.POSITIONING] = PositioningState.new()
states[BochaState.AIMING] = AimingState.new()
states[BochaState.THROWING] = ThrowingState.new()
states[BochaState.ROLLING] = RollingState.new()

# Cada ciclo de frame, delega al estado activo:
func _process(delta):
    states[current_state].update(delta)

func _physics_process(delta):
    states[current_state].physics_update(delta)

func _input(event):
    states[current_state].handle_input(event)

Por que usa call_deferred

En Godot, los hijos se inicializan antes que el padre. StateMachine._ready() se ejecuta antes que Bocha._ready(). Si el StateMachine intentara acceder a bocha.physics_body en su _ready() directo, seria null porque Bocha todavia no cacheo las referencias.

call_deferred("_init_states") programa la inicializacion para el siguiente frame, cuando todos los _ready() ya completaron.

Como funciona transition_to()

func transition_to(new_state: BochaState) -> void:
    # 1. Salir del estado actual (llama a exit())
    states[current_state].exit()

    # 2. Cambiar el estado activo
    current_state = new_state

    # 3. Entrar al nuevo estado (llama a enter())
    states[current_state].enter()

    # 4. Emitir señal para que el raiz se entere
    state_changed.emit(BochaState.keys()[current_state])

Como modificar

  • Agregar un nuevo estado: Crear un nuevo script que extienda RefCounted con los 5 metodos (init, enter, exit, update, physics_update, handle_input). Agregarlo al enum BochaState y crearlo con .new() en _init_states().
  • Cambiar el estado inicial: Modificar current_state = BochaState.X.
  • Agregar logging: Los prints ya estan en transition_to() y _input().

7. PositionController.gd (Movimiento Lateral)

Responsabilidad: Mover la bocha en el eje Z (izquierda/derecha) mientras esta en estado POSITIONING.

Logica

func update_position() -> void:
    # 1. Leer input de A/D o flechas
    var direction = 0.0
    if Input.is_action_pressed("move_left"): direction += 1.0
    if Input.is_action_pressed("move_right"): direction -= 1.0

    # 2. Calcular nueva posicion en Z
    var speed = bocha_root.lateral_speed * direction
    var new_z = bocha_root.physics_body.position.z + speed * get_process_delta_time()

    # 3. Respetar limites laterales
    new_z = clamp(new_z, bocha_root.lateral_bounds.x, bocha_root.lateral_bounds.y)

    # 4. Aplicar solo en Z, mantener X e Y
    var new_pos = bocha_root.physics_body.position
    new_pos.z = new_z
    bocha_root.physics_body.position = new_pos

Como modificar

  • Cambiar eje de movimiento: Cambiar position.z por position.x si tu escena usa ejes distintos.
  • Cambiar las teclas: Modificar "move_left" y "move_right" por otras acciones del Input Map.
  • Agregar aceleracion: En vez de mover directamente, acumular velocidad y aplicar con inercia.
  • Cambiar limites: Se controlan desde lateral_bounds en el nodo raiz Bocha.

8. ThrowController.gd (Gesto de Lanzamiento)

Responsabilidad: Detectar el gesto del mouse (click → arrastrar → soltar) y calcular la potencia y curva del lanzamiento.

Gesto: "Pull & Release" (estilo tirachinas)

Click DOWN
→
Arrastrar mouse
→
Click UP (soltar)
→
RELEASED

Como se calcula la potencia

# Distancia arrastrada (en pixeles de pantalla)
var delta = drag_current_pos - drag_start_pos
var drag_distance = delta.length()

# Normalizar entre 0.0 y 1.0
var normalized = clamp(drag_distance / max_drag_distance, 0.0, 1.0)

# Potencia final = normalizado * maxima potencia configurable
data.power = normalized * bocha_root.max_throw_power

Como se calcula la curva

# Componente X de la direccion del arrastre (lateral en pantalla)
var lateral = drag_direction.x

# Curva va de -1.0 a 1.0, multiplicada por sensibilidad
data.curve = clamp(lateral * sensitivity, -1.0, 1.0)

Variables de configuracion

VariableDefaultQue controla
min_drag_distance20.0Pixeles minimos de arrastre para considerar un lanzamiento valido
max_drag_distance400.0Pixeles de arrastre para potencia maxima (100%)
sensitivity1.0Multiplicador del efecto de curva

Como modificar

  • Hacer mas sensible: Bajar max_drag_distance a 200. Con menos arrastre se llega a potencia maxima.
  • Hacer menos sensible: Subir max_drag_distance a 600. Necesitas arrastrar mas.
  • Requerir arrastre minimo: Subir min_drag_distance para evitar lanzamientos accidentales con clicks cortos.
  • Mas curva: Subir sensitivity a 2.0 o 3.0.
  • Sin curva: Poner sensitivity = 0.0 o apply_curve = false en ThrowingState.
  • Cambiar direccion de lanzamiento: Modificar data.direction = Vector3(-1, 0, 0) en get_throw_data().

9. Los 4 Estados (Scripts de Estado)

Todos los estados extienden RefCounted (no son nodos). Comparten la misma interfaz de 5 metodos. Se crean con .new() en el StateMachine.

Interfaz comun de cada estado

MetodoCuando se llamaQue debe hacer
init(root)Una vez, al crear el estadoGuardar referencia al nodo raiz Bocha
enter()Cada vez que se entra al estadoConfigurar el entorno (freeze, resetear datos, mostrar UI)
update(delta)Cada frame (_process) mientras activoLogica no-fisica (input, UI, logica de juego)
physics_update(delta)Cada physics frame mientras activoLogica de fisica (chequear velocidad, aplicar fuerzas)
handle_input(event)Cada evento de input mientras activoProcesar input y decidir si transicionar
exit()Cada vez que se sale del estadoCleanup (ocultar UI, limpiar datos)

9.1 PositioningState.gd

Que hace: Permite mover la bocha lateralmente con A/D. Al hacer click, transiciona a AIMING.

enter(): Freeza el PhysicsBody (para que no caiga) y resetea velocidad a cero.

exit(): NO hace nada. No unfreeza porque el siguiente estado (AimingState) maneja su propio freeze. Si unfreezara aca, la bocha caeria por gravedad antes de que AimingState entre.

update(): Delega al PositionController el movimiento lateral.

handle_input(): Cualquier click izquierdo → transiciona a AIMING. No usa raycast.

9.2 AimingState.gd

Que hace: Detecta el gesto del mouse (click→arrastrar→soltar) y calcula potencia/curva.

enter(): Freeza el PhysicsBody, inicia el ThrowController, muestra indicador de potencia.

exit(): Oculta el indicador de potencia. NO llama a stop_aiming() porque ThrowingState necesita los datos del gesto.

update(): Llama a throw_controller.update_aim() para rastrear el arrastre. Si el gesto llego a RELEASED y la potencia es suficiente, transiciona a THROWING. Si la potencia es insuficiente, reinicia el gesto para intentar de nuevo.

handle_input(): Pasa todos los eventos al throw_controller.handle_input(event). Este es el canal por donde el ThrowController recibe los clicks del usuario.

9.3 ThrowingState.gd

Que hace: Aplica el impulso de lanzamiento al PhysicsBody. Dura solo 1-2 frames.

enter(): Resetea el flag throw_applied = false.

update(): Solo efectos visuales (particulas). NO aplica fisica aca.

physics_update():
  1. Unfreeza el PhysicsBody (freeze = false)
  2. Lee los datos del lanzamiento del ThrowController
  3. Aplica impulso central: apply_central_impulse(direction * power)
  4. Aplica curva: angular_velocity.y = curve * multiplier
  5. Emite señal bocha_thrown
  6. Transiciona a ROLLING


exit(): Llama a throw_controller.stop_aiming() para limpiar los datos del gesto.

9.4 RollingState.gd

Que hace: Monitorea la bocha mientras rueda. Cuando se detiene, vuelve a POSITIONING.

enter(): Resetea el timer de detencion, asegura que el PhysicsBody no este freezeado.

update(): Actualiza el trail de particulas segun la velocidad.

physics_update(): Chequea si la velocidad es menor a stop_threshold (0.05). Si lo es por mas de required_stop_time (0.5 segundos), detiene completamente la bocha y transiciona a POSITIONING.

handle_input(): Ignora todo input.

10. InputHandler.gd (Reservado)

Estado actual: Este script tiene un metodo is_clicking_on_bocha() que usa raycast para detectar si el click fue sobre la bocha. NO se usa actualmente porque PositioningState acepta cualquier click.

Para que sirve en el futuro: Si queres que solo se pueda iniciar el lanzamiento haciendo click directamente sobre la bocha (no en cualquier parte de la pantalla), descomenta el raycast en PositioningState y usalo.

Requisitos para que funcione: Necesita una camara activa en la escena y un CollisionShape3D con tamaño apropiado en el PhysicsBody.

12. Flujo Completo Paso a Paso

Este es el recorrido completo desde que se inicia la escena hasta que la bocha se detiene.

Fase 1: Inicializacion

# Frame 0:
1. Bocha._ready()
   → Cachea referencias a hijos
   → Inyecta bocha_root en controladores
   → Conecta señal state_changed

2. StateMachine._ready()
   → Programa call_deferred("_init_states")

3. call_deferred se ejecuta (StateMachine._init_states)
   → Crea 4 estados con .new()
   → Inicializa cada estado con init(bocha_root)
   → Entra a POSITIONING (llama a enter())
   → POSITIONING freeza el PhysicsBody

Fase 2: Posicionamiento

# Cada frame mientras en POSITIONING:
1. StateMachine._process() → PositioningState.update()
   → PositionController.update_position()
   → Lee A/D, mueve en Z

2. StateMachine._input() → PositioningState.handle_input()
   → Si click izquierdo → transition_to(AIMING)

# Transicion a AIMING:
3. PositioningState.exit() → no hace nada
4. AimingState.enter() → freeza PhysicsBody, inicia ThrowController

Fase 3: Apuntado (Gesto)

# Cada frame mientras en AIMING:
1. StateMachine._process() → AimingState.update()
   → ThrowController.update_aim()
   → Si DRAGGING: calcula distancia y direccion del arrastre
   → Si RELEASED: lee datos del lanzamiento
   → Si power >= min_power_to_throw → transition_to(THROWING)

2. StateMachine._input() → AimingState.handle_input()
   → ThrowController.handle_input(event)
   → Si mouse DOWN → DRAGGING, registra posicion inicial
   → Si mouse UP → RELEASED

# Transicion a THROWING:
3. AimingState.exit() → oculta indicador de potencia (NO limpia datos)

Fase 4: Lanzamiento

# Frame de THROWING:
1. ThrowingState.enter() → throw_applied = false

2. ThrowingState.update() → efectos visuales, throw_applied = true

3. ThrowingState.physics_update()
   → Si throw_applied: transiciona a ROLLING
   → Si no throw_applied:
     a. unfreeza PhysicsBody
     b. lee datos del ThrowController
     c. aplica impulso central
     d. aplica curva angular
     e. emite señal bocha_thrown
     f. throw_applied = true

# Transicion a ROLLING:
4. ThrowingState.exit() → stop_aiming() (limpia datos del gesto)

Fase 5: Rodado

# Cada physics frame mientras en ROLLING:
1. RollingState.physics_update()
   → Lee velocidad del PhysicsBody
   → Si velocidad < 0.05 por 0.5 segundos:
     a. Detiene completamente (velocity = 0, angular = 0)
     b. Emite señal bocha_stopped
     c. transition_to(POSITIONING)

2. RollingState.update()
   → Actualiza trail de particulas segun velocidad

14. Como Personalizar

Cambiar la velocidad de movimiento lateral

# En el inspector del nodo Bocha:
Lateral Speed: 5.0  # Mas rapido
Lateral Speed: 1.0  # Mas lento

# O en codigo, en bocha.gd:
@export var lateral_speed: float = 5.0

Cambiar los limites de movimiento lateral

# En el inspector del nodo Bocha:
Lateral Bounds: X = -3.0, Y = 3.0  # Mas amplio
Lateral Bounds: X = -1.0, Y = 1.0  # Mas estrecho

Cambiar la sensibilidad del lanzamiento

# En el inspector del nodo ThrowController:
Max Drag Distance: 200.0  # Mas sensible (menos arrastre = mas potencia)
Max Drag Distance: 600.0  # Menos sensible (mas arrastre necesario)
Min Drag Distance: 50.0   # Evita lanzamientos accidentales

Cambiar la potencia maxima del lanzamiento

# En el inspector del nodo Bocha:
Max Throw Power: 25.0  # Mas fuerte
Max Throw Power: 8.0   # Mas suave

Cambiar la friccion de la bocha

# En bocha.tscn, el PhysicsMaterial del PhysicsBody:
friction = 0.5  # Mas friccion, se detiene mas rapido
friction = 0.1  # Menos friccion, rueda mas lejos
bounce = 0.0    # Sin rebote
bounce = 0.5    # Con rebote

Cambiar cuando se considera "detenida"

# En RollingState.gd (o como @export):
@export var stop_threshold: float = 0.1    # Se detecta antes como detenida
@export var required_stop_time: float = 0.2 # Menos tiempo de espera

Cambiar la direccion de lanzamiento

# En ThrowController.gd, metodo get_throw_data():
# Si tu escena usa ejes distintos, cambia esta linea:
data.direction = Vector3(-1, 0, 0)  # -X = adelante

# Otras opciones:
data.direction = Vector3(0, 0, -1)  # -Z = adelante
data.direction = Vector3(1, 0, 0)   # +X = adelante

15. Problemas Comunes y Soluciones

La bocha no se mueve con A/D

Causa: Las acciones move_left y move_right no estan configuradas en el Input Map del proyecto.

Solucion: Ir a Project → Project Settings → Input Map. Agregar las acciones move_left (tecla A o flecha izquierda) y move_right (tecla D o flecha derecha).

La bocha cae al suelo inmediatamente

Causa: El PhysicsBody no esta freezeado durante POSITIONING o AIMING.

Verificar: En el Output, buscar [PositioningState] PhysicsBody freezeado. Si no aparece, el estado no esta entrando correctamente. Si aparece pero la bocha igual se mueve, verificar que freeze = true se este aplicando antes de que la fisica procese el frame.

El lanzamiento no tiene fuerza

Causa 1: El arrastre del mouse es muy corto (menos de min_drag_distance).

Solucion: Arrastrar mas lejos al hacer el gesto, o bajar min_drag_distance.

Causa 2: max_throw_power es muy bajo.

Solucion: Subir max_throw_power en el inspector del nodo Bocha.

La bocha nunca se detiene

Causa: La friccion del PhysicsMaterial es muy baja o el stop_threshold es muy bajo.

Solucion: Subir friction del PhysicsMaterial a 0.5 o mas, o subir stop_threshold en RollingState a 0.1.

El impulso es (0, 0, 0)

Causa: Los datos del gesto se resetearon antes de ser leidos. Esto pasaba cuando AimingState.exit() llamaba a stop_aiming().

Solucion: Ya esta fixeado. AimingState.exit() NO llama a stop_aiming(). ThrowingState.exit() lo llama despues de usar los datos. Si ves este problema de nuevo, verificar que AimingState.exit() no limpie los datos del gesto.

Los estados no se inicializan

Causa: call_deferred no se ejecuto o fallo.

Verificar: Buscar en el Output [StateMachine] Inicializando estados.... Si no aparece, el StateMachine no se inicializo. Verificar que el nodo StateMachine tenga el script StateMachine.gd asignado.
Resumen: Esta escena usa composicion pura con 7 nodos hijos + 4 estados como Resources. Cada componente tiene una responsabilidad unica. La comunicacion es por referencias inyectadas y señales. El flujo es: POSITIONING → AIMING → THROWING → ROLLING → POSITIONING (ciclo).
