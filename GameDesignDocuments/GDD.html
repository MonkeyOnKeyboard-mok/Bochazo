1. Conceptos Básicos de Godot que Necesitan Entender

Antes de empezar a crear escenas, es fundamental que todos entiendan estos conceptos. No necesitan ser expertos, pero sí tener claro cómo funciona Godot por dentro.

Escenas (.tscn)

En Godot TODO es una escena. Un menú es una escena, un botón es una escena, un personaje es una escena, e incluso una bocha individual es una escena.

Las escenas se pueden instanciar dentro de otras escenas. Esto es como "meter una escena dentro de otra".

Ejemplo: La escena bocce_ball.tscn se instancia 12 veces dentro de game_main.tscn (6 por jugador).

Nodos y Árboles

Cada escena está compuesta por nodos organizados en forma de árbol jerárquico. Un nodo padre puede tener varios hijos.

Cada tipo de nodo tiene una función específica: Node3D para objetos 3D, Control para UI, RigidBody3D para física, etc.

Regla de oro: Si un nodo hace más de 2 cosas diferentes, probablemente deberías dividirlo en nodos más pequeños.

Señales (Signals)

Las señales son la forma en que los nodos se comunican entre sí. Un nodo emite una señal y otro nodo la recibe y ejecuta una función.

Es como un sistema de "radio": alguien habla (emite) y quien está en el canal escucha (recibe).

Ventaja: Los nodos no necesitan conocerse entre sí. Solo necesitan conectarse a la misma señal.

Tipos de Nodos que Van a Usar

Nodo Para qué sirve Dónde lo van a usar
Node3D Nodo base para cualquier cosa en 3D GameManager, contenedores de objetos
RigidBody3D Objetos con física real (gravedad, colisiones, rebotes) Bochas, bochin
StaticBody3D Objetos sólidos que no se mueven Cancha, paredes, suelo
Area3D Zonas que detectan cuando algo entra/sale Límites de cancha, zonas de puntuación
Camera3D Cámara que renderiza la vista 3D Cámara del juego
Control Nodo base para interfaces 2D (UI) Menús, HUD, botones
VBoxContainer Organiza hijos verticalmente de forma automática Listas de botones en menús
AnimationPlayer Crea y reproduce animaciones de cualquier propiedad Transiciones, animaciones de UI
WorldEnvironment Configura iluminación global, cielo, efectos Escena principal del juego
DirectionalLight3D Luz direccional (como el sol) Iluminación de la cancha
Cómo funciona el flujo de trabajo en Godot
  1. Crear una escena → Agregás nodos al árbol de la escena
  2. Configurar propiedades → Desde el Inspector, ajustás tamaño, posición, colores, etc.
  3. Agregar scripts → Si necesitás lógica personalizada, le adjuntás un script .gd al nodo raíz
  4. Conectar señales → Desde el panel de señales del Inspector, conectás eventos a funciones
  5. Instanciar → Podés arrastrar una escena guardada dentro de otra escena
  6. Probar → F5 para ejecutar la escena principal, F6 para ejecutar la escena actual

Por qué esta estructura funciona bien para equipos
  • Cada carpeta = responsabilidad única: Si alguien trabaja en el menú, solo toca 01_main_menu/. Si otro trabaja en la física, solo toca components/court/ y components/balls/.
  • Componentes reutilizables: menu_button.tscn se usa en TODOS los menús. Si lo mejorás, mejora en todos lados.
  • Los recursos .tres están separados: El game designer puede ajustar las reglas del juego sin tocar código ni escenas.
  • Los managers están aislados: TurnManager, ScoreManager y BallManager no se conocen entre sí. Se comunican solo por señales.

3. Flujo del Juego (Diagrama Visual)

Este diagrama muestra cómo el jugador navega entre las distintas escenas del juego. Cada caja es una escena diferente.

flowchart TD A[Boot Scene] --> B[MainMenu Scene] B --> C{Game Mode?} C -->|1v1 Local| D[CharacterSelect Scene] C -->|Player vs IA| D C -->|Opciones| E[Options Scene] D --> F[Game Main Scene] E --> B F --> G{Round Complete?} G -->|Yes| H[RoundResults Scene] G -->|No| F H --> I{Match Complete?} I -->|No| D I -->|Yes| J[MatchResults Scene] J --> B F -->|Pause| K[PauseMenu Scene] K --> F K --> B
Cómo implementar este flujo paso a paso
  1. Empezar por Boot: Es la escena más simple. Solo carga y después cambia al MainMenu.
  2. Luego MainMenu: Crear los botones y que cada uno lleve a la escena correcta.
  3. CharacterSelect: Mostrar personajes y guardar la selección en game_config.gd (autoload).
  4. Game Main: Esta es la escena más compleja. Se construye por partes (ver roadmap).
  5. RoundResults / MatchResults: Pantallas que muestran información y tienen botones para continuar.
Tip: Cómo cambiar entre escenas en Godot

Usen get_tree().change_scene_to_file("res://scenes/01_main_menu/main_menu.tscn") para cambiar de escena. Para transiciones suaves, usen un CanvasLayer con un ColorRect negro que haga fade in/out con un AnimationPlayer.

4. Sistema de Turnos (La Lógica Central del Juego)

Este es el corazón del juego. Entender bien cómo funcionan los turnos es fundamental antes de empezar a programar.

flowchart TD Start[Inicio de Ronda] --> A[Player 1 lanza Bochin] A --> B[Player 1 lanza 1ra bocha] B --> C{Medir distancias al bochin} C --> D{P1 tiene bocha más cercana?} D -->|Sí| E[Turno: Player 2] D -->|No| F[Turno: Player 1] E --> G[Player 2 lanza bocha] F --> H[Player 1 lanza bocha] G --> I{Medir distancias} H --> I I --> J{Quién tiene la más cercana?} J -->|P1| K{P2 tiene bochas restantes?} J -->|P2| L{P1 tiene bochas restantes?} K -->|Sí| M[Turno: Player 2] K -->|No| N[Turno: Player 1 lanza las que le quedan] L -->|Sí| O[Turno: Player 1] L -->|No| P[Turno: Player 2 lanza las que le quedan] M --> G N --> Q[Player 1 lanza hasta completar 6] O --> H P --> R[Player 2 lanza hasta completar 6] Q --> S[Ambos completaron 6?] R --> S S -->|No| I S -->|Sí| T[Fin de Ronda → Scoring] style Start fill:#9f7aea,color:#fff style T fill:#f6ad55,color:#fff style S fill:#48bb78,color:#fff
Regla fundamental: El jugador que NO tiene la bocha más cercana al bochin es quien lanza. Esto es lo que hace que el juego sea estratégico: siempre el que va perdiendo tiene la oportunidad de mejorar.

Árbol de Nodos del TurnManager

El TurnManager es el nodo que controla toda esta lógica. Así debería verse su estructura:

TurnManager (Node) ├─ script: turn_manager.gd ├─ current_player: int (1 o 2) ├─ balls_remaining_p1: int (empieza en 6) ├─ balls_remaining_p2: int (empieza en 6) ├─ bochin_placed: bool ├─ round_active: bool │ ├─ Señales que EMITE: │ ├─ turn_changed(player_number) │ ├─ ball_thrown(player_number, ball_count) │ ├─ bochin_placed(position) │ ├─ round_ended(winner, scores) │ └─ player_must_throw(player_number) │ └─ Señales que RECIBE (conecta desde otros nodos): ├─ ball_stopped(position) ← desde BallManager └─ new_round_started() ← desde GameManager
Cómo implementar el TurnManager paso a paso
  1. Crear la escena: Nuevo nodo Node → guardar como turn_manager.tscn → adjuntar script.
  2. Definir variables: current_player, balls_remaining_p1, balls_remaining_p2, bochin_placed.
  3. Definir señales: En el script, usar signal turn_changed(player) al inicio del archivo.
  4. Función start_round(): Resetea variables, pone current_player = 1, emite turn_changed.
  5. Función on_ball_stopped(): Se conecta a la señal de BallManager. Mide distancias, decide quién tira, emite player_must_throw.
  6. Función switch_turn(): Cambia current_player, verifica si alguien se quedó sin bochas, emite turn_changed.
  7. Función check_round_end(): Si ambos tienen 0 bochas, emite round_ended.

5. Sistema de Puntuación

Al final de cada ronda, se calculan los puntos. El algoritmo es sencillo pero hay que entenderlo bien.

flowchart LR A[Todas las bochas lanzadas] --> B[Calcular distancia de cada bocha al bochin] B --> C[Ordenar lista de menor a mayor distancia] C --> D[Identificar dueño de la primera bocha P1 o P2] D --> E[Contar bochas consecutivas del mismo dueño] E --> F[Puntos = cantidad de bochas consecutivas] F --> G[Asignar puntos al jugador ganador] G --> H{Alguien llegó a 12 puntos?} H -->|No| I[Nueva ronda - Ganador comienza] H -->|Sí| J[Fin del Match] style A fill:#667eea,color:#fff style F fill:#f6ad55,color:#fff style J fill:#48bb78,color:#fff
Ejemplo práctico:

Lista ordenada por distancia al bochin:
1. P1_Bocha3 - 0.5m ← Líder
2. P1_Bocha1 - 0.8m ← +1 punto
3. P1_Bocha5 - 1.2m ← +1 punto
4. P2_Bocha2 - 1.5m ← CORTE (cambia de jugador)

Resultado: P1 gana 3 puntos
Tip: Cómo calcular distancias en Godot

Usen bocha.global_position.distance_to(bochin.global_position) para obtener la distancia en metros entre una bocha y el bochin. Es una función built-in de Godot.

6. Sistema de Lanzamiento por Gestos

Esta es la mecánica principal del juego. El jugador hace un gesto con el mouse que se traduce en fuerza, dirección y efecto de la bocha.

sequenceDiagram participant P as Jugador participant GT as GestureTracker participant TH as ThrowHandler participant BB as BocceBall (RigidBody3D) participant HUD as GameHUD participant CAM as Camera P->>GT: Click Izquierdo sobre la bocha GT->>HUD: Mostrar barra de potencia GT->>GT: Empezar a registrar posición del mouse P->>GT: Mueve mouse hacia ABAJO Note over GT: Fase 1: "Toma de carrera"
acumula potencia según distancia GT->>HUD: Actualizar barra de potencia P->>GT: Mueve mouse hacia ARRIBA (recto o curvo) Note over GT: Fase 2: Dirección y efecto
calcula vector y curva GT->>HUD: Mostrar preview de trayectoria P->>GT: Suelta el click (Mouse Up) GT->>TH: Enviar datos del gesto
(potencia, dirección, curva) TH->>TH: Calcular impulso y velocidad angular TH->>BB: apply_central_impulse(fuerza) TH->>BB: angular_velocity = efecto/curva TH->>CAM: Shake suave al lanzar BB->>BB: Simulación física (rodamiento + fricción) BB->>TH: velocity < umbral (se detuvo) TH->>HUD: Ocultar barra de potencia TH->>TH: Emitir señal ball_stopped

Árbol de Nodos del ThrowSystem

ThrowSystem (Node3D) ├─ GestureTracker (Node) │ ├─ script: gesture_tracker.gd │ ├─ Detecta: mouse_down, mouse_move, mouse_up │ ├─ Registra: posiciones del mouse en un array │ └─ Emite: gesture_completed(power, direction, curve) │ ├─ ThrowHandler (Node) │ ├─ script: throw_handler.gd │ ├─ Recibe: datos del gesto │ ├─ Calcula: vector de impulso y velocidad angular │ ├─ Referencia: a la bocha activa (RigidBody3D) │ └─ Ejecuta: ball.apply_central_impulse() + angular_velocity │ └─ TrajectoryPreview (MeshInstance3D - opcional) └─ Muestra línea punteada de hacia dónde va la bocha
Cómo implementar el sistema de gestos paso a paso
  1. GestureTracker: Conectar señales gui_input del botón/bocha. En _input(event), registrar event.position cuando es InputEventMouseMotion.
  2. Fase 1 (abajo): Si el mouse se mueve hacia abajo desde el punto inicial, acumular "potencia" proporcional a la distancia recorrida.
  3. Fase 2 (arriba): Cuando el mouse cambia de dirección y va hacia arriba, registrar la trayectoria. Si va recto = sin efecto. Si hace curva = efecto lateral.
  4. Mouse Up: Calcular el vector final: power = magnitud, direction = ángulo horizontal, curve = desviación lateral.
  5. ThrowHandler: Recibe los datos y aplica apply_central_impulse(Vector3) a la bocha. La curva se aplica con angular_velocity.
  6. Barra de potencia: Un ProgressBar en el HUD que se actualiza en tiempo real mientras el jugador mueve el mouse hacia abajo.
Cuidado con esto
  • La bocha es un RigidBody3D. NO muevan la bocha con position o translate. Solo usen apply_central_impulse() o apply_force().
  • Configuren bien la linear_damp y angular_damp en la bocha para que frene naturalmente. Valores típicos: linear_damp = 0.8, angular_damp = 0.9.
  • Para detectar que la bocha se detuvo: chequear en _physics_process si linear_velocity.length() < 0.1.

Tip: CanvasLayer para UI sobre 3D

El CanvasLayer hace que la UI se renderice POR ENCIMA del mundo 3D, sin importar dónde esté la cámara. Siempre usen un CanvasLayer como padre de cualquier HUD o menú que aparezca durante el juego 3D.

8. Cómo Conectar las Señales (Comunicación entre Nodos)

Las señales son la forma en que los nodos se comunican. En Godot pueden conectarse de dos formas: desde el editor (visual) o desde código. Para este proyecto, recomendamos una mezcla de ambas.

El Bus Central de Señales (GlobalSignals)

Para que los nodos no necesiten conocerse entre sí, usamos un SignalBus (autoload). Es un script que está disponible en TODAS las escenas y actúa como intermediario.

# global_signals.gd (autoload) # Este script NO tiene nodos, es solo un script en Autoload # Señales que cualquier nodo puede emitir: signal game_started(mode) signal character_selected(player, character_id) signal ball_thrown(player, ball_number) signal ball_stopped(ball_reference) signal turn_changed(player_number) signal round_ended(winner, scores) signal match_ended(winner) signal game_paused() signal game_resumed() # Cualquier nodo puede: # GlobalSignals.turn_changed.emit(2) # Y otro puede: # GlobalSignals.turn_changed.connect(_on_turn_changed)
Cuándo usar señales directas vs GlobalSignals
  • Señales directas: Cuando un nodo hijo le habla a su padre, o entre nodos que están en la misma escena. Ejemplo: un botón que le avisa a su menú que fue presionado.
  • GlobalSignals: Cuando nodos de escenas diferentes necesitan comunicarse. Ejemplo: BallManager (en game_main) le avisa a GameHUD (también en game_main) que una bocha se detuvo.
  • Regla práctica: Si los nodos están en el mismo árbol de escena → señal directa. Si están en árboles diferentes → GlobalSignals.

Cómo conectar señales desde el editor de Godot

Paso a paso visual
  1. Seleccioná el nodo que emite la señal (ej: un Button).
  2. En el panel derecho, ir a la pestaña "Node" (al lado de "Inspector").
  3. Hacer doble click en la señal que querés usar (ej: pressed).
  4. Se abre un diálogo: elegir el nodo que recibe la señal.
  5. Godot genera automáticamente la función conectada en el script.
  6. Completar la función con lo que querés que pase.

Cómo crear y usar recursos .tres en Godot
  1. Crear una clase que extienda Resource (como las de arriba).
  2. En el editor de Godot: click derecho en el FileSystem → New Resource → buscar la clase (ej: GameRules).

10. Estadísticas de Personajes (Game Design)

Las estadísticas hacen que cada personaje se sienta diferente. Estas son las que recomendamos para un juego de bochas:

Stat Qué afecta Valor bajo (1-3) Valor alto (8-10)
Potencia Fuerza máxima del lanzamiento Lanzamientos suaves, difícil llegar al fondo Puede lanzar muy fuerte, mayor rango
Precisión Qué tan fielmente la bocha sigue la dirección del gesto La bocha se desvía más del gesto La bocha va exactamente donde apuntó el gesto
Control de Efecto Qué tanto impacto tiene la curva del gesto en la trayectoria Los gestos curvos tienen poco efecto Puede hacer curvas pronunciadas
Consistencia Qué tan repetibles son sus lanzamientos (varianza aleatoria) Cada lanzamiento es impredecible Siempre lanza de forma similar
Cómo implementar las stats en el juego
  • Potencia: Multiplicar la fuerza del impulso: final_power = base_power * (power / 5.0)
  • Precisión: Agregar un ángulo aleatorio de desviación: desviación = (10 - accuracy) * 0.5 grados
  • Control de Efecto: Multiplicar la velocidad angular: spin = curve * (spin_control / 5.0)
  • Consistencia: Agregar variación aleatoria a todos los cálculos: varianza = (10 - consistency) * 0.02

15. Recomendaciones Finales y Buenas Prácticas

Principios de Diseño para este Proyecto
  1. Una escena = una responsabilidad: Si una escena hace más de 3 cosas diferentes, dividirla en escenas más pequeñas.
  2. Composición sobre herencia: Instanciar escenas dentro de otras, NO heredar scripts.
  3. Señales sobre referencias directas: Usar GlobalSignals para comunicación entre sistemas que no se conocen.
  4. Recursos sobre hardcode: Todos los valores de juego (reglas, física, stats) en archivos .tres.
  5. Nodos exportables: Usar @export para configurar desde el Inspector en vez de hardcodear valores.

Convenciones de Nomenclatura

Tipo Convención Ejemplo
Escenas snake_case.tscn character_select.tscn
Scripts snake_case.gd turn_manager.gd
Recursos snake_case.tres physics_config.tres
Nodos PascalCase TurnManager, BallSpawner
Señales snake_case turn_changed, ball_thrown
Variables snake_case current_player, max_balls
Constantes UPPER_SNAKE_CASE MAX_BALLS_PER_PLAYER

Errores Comunes a Evitar

No hagan esto
  • No muevan RigidBody3D con position: Usen apply_central_impulse() o apply_force().
  • No pongan toda la lógica en un solo script: Dividan en managers, componentes y sistemas.
  • No hardcodeen valores: Usen recursos .tres para todo lo configurable.
  • No referencien nodos por ruta absoluta: Usen señales o @export para referencias.
  • No ignoren los warnings del editor: Godot les avisa cuando algo está mal configurado.
  • No trabajen todos sobre la misma rama: Cada feature en su rama, mergear solo cuando funciona.

Optimizaciones Recomendadas

  • Usar VisibilityNotifier3D para desactivar objetos fuera de cámara
  • Implementar object pooling para las bochas (reutilizar en vez de instanciar/destruir)
  • Pre-cargar escenas pesadas en boot.tscn o durante transiciones
  • Usar call_deferred() para operaciones pesadas en señales
  • Usar ResourceLoader.load_threaded_request() para carga asíncrona
