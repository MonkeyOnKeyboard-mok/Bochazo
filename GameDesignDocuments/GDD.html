flowchart TD Start[Inicio de Ronda] --> A[Player 1 lanza Bochin] A --> B[Player 1 lanza 1ra bocha] B --> C{Medir distancias} C --> D[¿P1 tiene bocha más cercana?] D -->|Sí| E[Turno: Player 2] D -->|No| F[Turno: Player 1] E --> G[Player 2 lanza bocha] F --> H[Player 1 lanza bocha] G --> I{Medir distancias} H --> I I --> J{¿Quién tiene la más cercana?} J -->|P1| K{P2 tiene bochas?} J -->|P2| L{P1 tiene bochas?} K -->|Sí| M[Turno: Player 2] K -->|No| N[Turno: Player 1 x restantes] L -->|Sí| O[Turno: Player 1] L -->|No| P[Turno: Player 2 x restantes] M --> G N --> Q[Player 1 lanza hasta completar 6] O --> H P --> R[Player 2 lanza hasta completar 6] Q --> S[Ambos completaron 6?] R --> S S -->|No| I S -->|Sí| T[Fin de Ronda - Scoring] style Start fill:#9f7aea style T fill:#f6ad55 style S fill:#48bb78
Regla clave: El jugador que NO tiene la bocha más cercana al bochin es quien lanza. Si un jugador agota sus 6 bochas, el otro jugador puede lanzar todas las que le queden.

sequenceDiagram participant P as Player participant I as InputHandler participant G as GestureTracker participant T as ThrowSystem participant B as Ball (RigidBody) participant C as Camera participant H as HUD P->>I: Click Izquierdo en Ball I->>H: Show power bar I->>G: Start tracking P->>G: Mouse Down + Move Down Note over G: Fase 1: "Toma de carrera"
acumula potencia G->>H: Update power bar P->>G: Move Up (straight/curve) Note over G: Fase 2: Dirección y efecto
calcula vector y curva G->>H: Show trajectory preview P->>I: Mouse Up (Release) I->>G: Stop tracking G->>T: Send gesture data
(power, direction, curve) T->>T: Calculate physics
impulse & angular_velocity T->>B: apply_central_impulse() T->>B: angular_velocity = curve T->>C: Shake camera slightly B->>B: Physics simulation Note over B: Ball rolls with friction
and damping B->>T: velocity < threshold T->>H: Hide power bar T->>H: Show "Ball Stopped" Note over T: Signal: ball_stopped

Reglas de Git:
  • Nunca hacer commit directo a main excepto el Tech Lead
  • Cada feature debe tener su propia rama: feat/nombre-feature
  • Las ramas de features NO se tocan entre sí. Cada una trabaja en escenas/scripts distintos
  • Usar dev/integration para pruebas de integración antes de mergear a main
  • Commits atómicos: un cambio = un commit descriptivo

💡 Recomendaciones Finales

Principios de Diseño:
  1. Una escena = una responsabilidad: Si una escena hace más de 3 cosas, dividirla
  2. Composición sobre herencia: Usar instancia de escenas, no herencia de scripts
  3. Señales sobre referencias directas: Usar SignalBus para comunicación entre sistemas
  4. Recursos sobre hardcode: Todos los valores de juego en archivos .tres
  5. Nodos exportables: Usar @export para configurar desde el Inspector

Tipo Convención Ejemplo

Optimizaciones Recomendadas

  • Usar VisibilityNotifier3D para desactivar objetos fuera de cámara
  • Implementar object pooling para las bochas (reutilizar en vez de instanciar/destruir)
  • Usar LOD (Level of Detail) en modelos 3D si es necesario
  • Pre-cargar escenas pesadas en boot.tscn o durante transiciones
  • Usar call_deferred() para operaciones pesadas en señales
