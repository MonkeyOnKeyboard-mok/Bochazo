Game Config Throw · Court · Bocce · Bochin

Visión General

Cada entidad del juego (Throw, Court, Bocce, Bochin) tiene su propia configuración via Resource. Esto permite:

  • Crear diferentes personajes con distinta fuerza, curva y precisión
  • Crear diferentes canchas con distinta fricción, tamaño y obstáculos
  • Crear diferentes tipos de bola con distinta masa y rebote
  • Ajustar todo desde el inspector sin tocar código

PhysicsConfig RESOURCE

Física de las bolas. Actualmente compartido entre BocceBall y Bochin, pero deberían tener configuraciones separadas porque el Bochin es más pequeño y liviano.

Problema actual: bocce_ball.tres tiene linear_damping = 0.1 pero physics_config.gd define linear_damping = 0.8. El .tres no tiene ball_mass, ball_friction, ni ball_bounce definidos (usa los defaults del script). Además, el Bochin usa el mismo .tres que la BocceBall.

Bochin (propuesto)

Diferencia clave

El Bochin es más liviano (0.4 vs 1.0), más bouncy (0.5 vs 0.3) y se frena más rápido (damping 0.6 vs 0.8). Esto lo hace más sensible a los impactos pero más fácil de desplazar.

Código — Court (propuesto, agregando CourtConfig) 
extends StaticBody3D
class_name Court

@export var court_config: CourtConfig

func _ready():
    _apply_config()

func _apply_config():
    if not court_config: return
    var mat = PhysicsMaterial.new()
    mat.friction = court_config.court_friction
    mat.bounce = court_config.court_bounce
    
    # Aplicar a todas las CollisionShape3D hijas
    for child in get_children():
        if child is CollisionShape3D:
            # El shape hereda el material del StaticBody
            pass
    
    physics_material_override = mat

La cancha aplica PhysicsMaterial (friction + bounce) como override del StaticBody3D. Las bolas interactúan con esta superficie a través de la física de Godot.

Cómo interactúan las físicas

Cada bola tiene su propio PhysicsMaterial (friction + bounce). La cancha también tiene su PhysicsMaterial. Cuando una bola toca la cancha, Godot calcula la fricción y el rebote combinando ambos materiales.

Friction =media armónica de los dos materiales. Bounce = multiplicación. Así que:

SuperficieCourt FrictionCourt BounceBocceBall se frena en...BocceBall rebota en...
Tierra0.90.1Muy rápidoCasi nada
Césped0.70.2NormalPoco
Sintético0.40.4LentoAlgo
Hielo0.10.1Muy lentoCasi nada
Para que la bola llegue al fondo: El launch_angle_deg (3°) le da un pequeño salto inicial. Pero la fricción de la cancha determina cuánto se frena. Si la cancha tiene poca fricción (sintético, hielo), la bola llega más lejos. Si tiene mucha (tierra), hay que usar más fuerza.

BocceBall y Bochin — Código actual

BocceBall — Escena + Script

bocce_ball.gd

extends RigidBody3D
class_name BocceBall

signal stopped_moving(ball_ref: BocceBall)
signal collision_occurred(impulse: Vector3)

@export_group("Configuración")
@export var physics_config: PhysicsConfig
@export var stop_velocity_threshold: float = 0.1
@export var debug_verbose: bool = false

@onready var ball_mesh: MeshInstance3D = $BallMesh
@onready var ball_collision: CollisionShape3D = $BallCollision

var _is_stopped: bool = false

func _ready():
    _apply_physics()
    body_entered.connect(_on_body_entered)

func _physics_process(_delta):
    if _is_stopped: return
    if linear_velocity.length() < stop_velocity_threshold:
        _is_stopped = true
        freeze = true
        stopped_moving.emit(self)
        if debug_verbose: print("[BocceBall] Se detuvo")
        freeze = false

func _apply_physics():
    if physics_config:
        mass = physics_config.ball_mass
        linear_damp = physics_config.linear_damping
        angular_damp = physics_config.angular_damping
        var phys_mat = PhysicsMaterial.new()
        phys_mat.friction = physics_config.ball_friction
        phys_mat.bounce = physics_config.ball_bounce
        physics_material_override = phys_mat
    else:
        push_warning("[BocceBall] physics_config no asignado.")

func _on_body_entered(body: Node):
    collision_occurred.emit(linear_velocity)
    if debug_verbose and linear_velocity.length() > 0.5:
        print("[BocceBall] Colisión con %s" % body.name)

bocce_ball.tscn (estructura)

BocceBall (RigidBody3D) ├── BallMesh (MeshInstance3D, visible=false) ├── BallCollision (CollisionShape3D, SphereShape3D) └── BochaAzul (MeshInstance3D, modelo importado)

Nota: Tiene un SphereMesh oculto + el modelo importado (BochaAzul) visible. El SphereShape3D no tiene radio definido en el .tscn — usa el default (1.0).

Bochin — Escena + Script

bochin.gd

extends BocceBall
class_name Bochin

Herencia directa de BocceBall. Solo hereda comportamiento, no tiene overrides propios.

bochin.tscn (estructura)

Bochin (RigidBody3D, mass=0.4) ├── BallMesh (MeshInstance3D, SphereMesh r=0.2) └── BallCollision (CollisionShape3D, SphereShape3D r=0.2)
Problema: El Bochin usa el mismo bocce_ball.tres que la BocceBall. Hereda los mismos valores de masa, fricción y rebote, pero los sobrescribe parcialmente en la escena (mass = 0.4, radio más chico). Debería tener su propio .tres con valores diferenciados.
Court — Escena
Court (StaticBody3D) ├── CanchaBochas (MeshInstance3D, visible=false) ├── CollisionShape3D (ConcavePolygonShape3D) └── CanchaBochas2 (instancia GLB, visible)

La cancha es un StaticBody3D con forma cóncava. No tiene PhysicsMaterial configurado — usa los defaults de Godot (friction 1.0, bounce 0.0). Necesita un CourtConfig resource.

Código Actual vs Propuesto

Problemas actuales

PhysicsConfig duplica ThrowConfig

Los campos min_throw_power, max_throw_power, power_multiplier, spin_multiplier, curve_sensitivity se superponen con ThrowConfig. Deben eliminarse.

Bochin usa config de BocceBall

El Bochin referencia bocce_ball.tres pero tiene masa y radio distintos. Necesita su propio bochin.tres.

Court sin PhysicsMaterial

El StaticBody3D de la cancha no tiene PhysicsMaterial. La fricción depende de los defaults de Godot. Necesita CourtConfig.

BocceBall stop_velocity_threshold hardcoded

El umbral de detención (0.1) está como @export en BocceBall. Debería estar en PhysicsConfig.

Flujo: Cada .tres se assigna en el editor. ThrowHandler distribuye los valores a los componentes en _ready(). ThrowHandler no necesita conocer los detalles de PhysicsConfig o CourtConfig — esas se asignan directamente a las bolas y la cancha.
