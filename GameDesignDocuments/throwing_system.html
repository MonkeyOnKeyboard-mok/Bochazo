Throw System

Sistema de lanzamiento de bochas — arquitectura, lógica y flujo de datos

Visión General

El sistema convierte un gesture del mouse (click & drag) en un lanzamiento 3D de una bocha con dirección, potencia y efecto (spin). Se divide en cinco componentes desacoplados conectados por señales.

graph LR
  subgraph Entrada
    A[GestureTracker] -->|gesture_started| B[GesturePower]
    A -->|gesture_dragging| B
    A -->|flick_detected| C[GestureDirection]
    A -->|gesture_ended| D[GestureSpin]
  end

  B -->|ProgressBar.value| E[ThrowHandler]
  C -->|direction_calculated| E
  D -->|spin_profile_calculated| E

  E -->|apply_central_impulse| F[RigidBody3D: BocceBall]
  E -->|linear_velocity.rotated| F

El flujo es unidireccional: de la entrada del usuario hacia la física. Un solo GestureTracker alimenta los tres módulos de cálculo, y ThrowHandler orquesta el resultado final.

Componentes — Lógica Detallada

GestureTracker

Archivo: gesture_tracker.gd · Nodo base: Node

Captura la entrada cruda del mouse y emite señales para los demás módulos.

Señales

gesture_started(start_pos: Vector2) gesture_dragging(current: Vector2, start: Vector2) flick_detected(dir: Vector2) gesture_ended(points: PackedVector2Array)

Máquina de estados implícita

stateDiagram-v2
  [*] --> Idle
  Idle --> Tracking : click izquierdo (pressed)
  Tracking --> Tracking : mouse motion
  Tracking --> Flick : dy pasa de +2 a -2
  Tracking --> Idle : soltar click (released)
  Flick --> Idle : soltar click (released)

  note right of Tracking: acumula points[]\nactualiza DebugLine\nemite gesture_dragging
  note right of Flick: emite flick_detected\n(con la dirección normalizada)

Detección de Flick

El flick se detecta cuando el usuario baja y luego sube rápidamente durante el mismo drag:

  1. Se marca was_moving_down = true cuando dy > 2 px (el cursor baja).
  2. Cuando was_moving_down es true y dy < -2 px (el cursor sube), se emite flick_detected con Vector2(pos.x - prev_pos.x, dy).normalized().
  3. Se usa el flag flick_emitted para emitir solo una vez por gesto.

Por qué importa: El flick representa el "latigazo" del lanzamiento. La dirección del flick alimenta directamente a GestureDirection.

Variables clave

VariableTipoPropósito
pointsPackedVector2ArrayTodos los puntos del gesto actual
prev_posVector2Posición del frame anterior (para calcular dy)
was_moving_downboolSe bajó en algún momento del gesto
flick_emittedboolEvita múltiples flicks por gesto

GesturePower

Archivo: gesture_power.gd · Nodo base: Node

Convierte la distancia vertical del drag en un porcentaje de potencia (0–100).

Lógica

flowchart TD
  A[gesture_started] --> B["bar.value = 0\npeak_value = 0"]
  C[gesture_dragging] --> D["current = clamp((cur.y - start.y) / max_drag_px, 0, 1) * 100"]
  D --> E["peak_value = max(peak_value, current)"]
  E --> F["bar.value = peak_value"]

La potencia solo sube, nunca baja. El jugador arrastra hacia abajo y la barra sube; si suelta un poco y arrastra menos, la barra se queda en el máximo alcanzado. Esto modela el esfuerzo acumulado de un lanzamiento real.

ParámetroDefaultDescripción
max_drag_px300Píxeles de drag vertical = 100% potencia

Observación: La dirección del drag es solo hacia abajo (Y positivo en pantalla). Si el usuario arrastra hacia arriba, el valor queda en 0. Esto asume que el gesto natural es tirar hacia abajo para "cargar" el lanzamiento.

GestureDirection

Archivo: gesture_direction.gd · Nodo base: Node

Traduce la dirección del flick a una dirección 2D y muestra una preview visual.

Lógica

  1. Se conecta a flick_detected del tracker.
  2. Al recibir el vector de dirección, dibuja una línea invertida en PreviewLine: dir * -preview_length apuntando desde el origen hacia donde "tirarías".
  3. Emite direction_calculated(dir: Vector2).

La inversión visual es porque en pantalla el flick va hacia abajo, pero la bola va hacia adelante — la línea preview apunta en la dirección opuesta al gesto, mostrando hacia dónde saldrá la bola.

ParámetroDefaultDescripción
preview_length150Largo en px de la línea de preview

GestureSpin

Archivo: gesture_spin.gd · Nodo base: Node

Calcula el perfil de desviación lateral del gesto — cuánto se curvó el trazo a cada lado.

Algoritmo

flowchart TD
  A[gesture_ended: points] --> B{points.size < 3?}
  B -->|Sí| C["emit spin_profile_calculated([])"]
  B -->|No| D["dir = (end - start).normalized()"]
  D --> E["perp = Vector2(-dir.y, dir.x)"]
  E --> F["Para cada punto p:"]
  F --> G["lateral = (p - start) · perp / sensitivity"]
  G --> H["clamp(lateral, -1, 1)"]
  H --> I["emit spin_profile_calculated(profile)"]

El algoritmo proyecta cada punto del gesto sobre el eje perpendicular a la dirección general del trazo. El resultado es un array de valores entre -1 y 1 que describe cómo se curva el gesto a lo largo de su recorrido.

Intuición: Si el gesto es una línea recta, todos los valores son cercanos a 0 (sin efecto). Si el gesto se curva hacia un lado, los valores suben hacia +1; si se curva hacia el otro, bajan hacia -1.

ParámetroDefaultDescripción
sensitivity100Divisor de la desviación lateral. Menor = efecto más amplio por menos curva.

ThrowHandler

Archivo: throw_handler.gd · Nodo base: Node

Orquestador final: recibe dirección, potencia y perfil de spin, y los aplica al RigidBody3D de la bocha.

Conexiones de señales

flowchart LR
  TK[GestureTracker] -->|gesture_ended| TH[ThrowHandler._on_throw]
  DM[GestureDirection] -->|direction_calculated| TH
  SM[GestureSpin] -->|spin_profile_calculated| TH

  TH -->|_physics_process| RIGID["RigidBody3D.linear_velocity"]

Cálculo de dirección 3D

Cuando GestureDirection emite una dirección 2D (dx, dy), ThrowHandler la convierte a 3D:

flowchart TD
  A["Vector2(dir.x, dir.y) del flick"] --> B["raw = Vector3(-dir.y, 0, dir.x)"]
  B --> C["raw.x = max(raw.x, 0.01)  // siempre hacia adelante"]
  C --> D["limit_z = raw.x * max_tangent"]
  D --> E["raw.z = clamp(raw.z, -limit_z, limit_z)"]
  E --> F["dir = raw.normalized()"]

El mapeo convierte las coordenadas de pantalla al espacio 3D:

  • -dir.y (pantalla) → componente X (3D): hacia adelante en la pista. Se fuerza ≥ 0.01 para que siempre haya componente frontal.
  • dir.x (pantalla) → componente Z (3D): izquierda/derecha.
  • El eje Y (3D) es siempre 0: el lanzamiento es plano.

El cono de tiro limita la desviación lateral: max_tangent = tan(max_spread_deg). La componente Z nunca puede superar X * max_tangent, creando un cono angular que impide que la bocha se escape de la pista.

Ejecución del lanzamiento

flowchart TD
  A[gesture_ended] --> B{power_bar existe?}
  B -->|No| Z[return]
  B -->|Sí| C["power = clamp(bar.value / 100, 0, 1)"]
  C --> D{power < 0.05?}
  D -->|Sí| Z
  D -->|No| E["ball.apply_central_impulse(dir * power * max_force)"]
  E --> F{spin_profile no vacío?}
  F -->|No| G["is_applying_spin = false"]
  F -->|Sí| H["avg_dev = promedio(abs(values))"]
  H --> I{avg_dev > min_curve_threshold?}
  I -->|No| G
  I -->|Sí| J["is_applying_spin = true\nspin_idx = 0\nspin_timer = 0.0"]

Umbral de curvatura mínima (min_curve_threshold): filtra gestos que son prácticamente rectos. Si el promedio de desviación lateral es menor al umbral, no se aplica spin — evitando micro-temblores o gestos puramente verticales.

Aplicación del Spin en tiempo de ejecución

Durante _physics_process, si is_applying_spin es true:

flowchart TD
  A[_physics_process] --> B{is_applying_spin && spin_profile no vacío?}
  B -->|No| Z[return]
  B -->|Sí| C["spin_timer += delta"]
  C --> D{spin_timer >= 1/spin_hz?}
  D -->|No| FIN[esperar]
  D -->|Sí| E["spin_timer = 0"]
  E --> F{spin_idx < spin_profile.size?}
  F -->|No| G["is_applying_spin = false"]
  F -->|Sí| H["curve = spin_profile[spin_idx]"]
  H --> I["steer_angle = curve * steer_strength * delta"]
  I --> J{vel.length > 0.1?}
  J -->|No| K["spin_idx += 1"]
  J -->|Sí| L["ball.linear_velocity = vel.rotated(Vector3.UP, steer_angle)"]
  L --> K

El spin funciona rotando el vector velocidad sin agregar energía:

  • Cada tick (a frecuencia spin_hz), se consume un valor del perfil.
  • El valor (-1 a 1) se multiplica por steer_strength y delta, generando un ángulo de rotación.
  • Se rota linear_velocity alrededor de Vector3.UP por ese ángulo.
  • Esto curva la trayectoria sin cambiar la velocidad — simula el efecto Magnus.

Frecuencia spin_hz: Con el valor default de 8 Hz, la curva es progresiva y suave. En el test usa 1000 Hz (casi cada frame), lo que hace que la curva responda inmediatamente al perfil.

GestureTracker es el único productor de eventos. Los tres módulos de procesamiento (GesturePower, GestureDirection, GestureSpin) son consumidores puros y no se comunican entre sí. ThrowHandler es el único consumidor final — no emite señales.

Restricciones y Conos de Seguridad

graph TD
  A["max_tangent = tan(5°) ≈ 0.0875"] --> B["limit_z = raw.x * max_tangent"]
  B --> C["Z nunca supera X * 8.75%"]
  C --> D["La bocha nunca supera ~5° de desviación lateral"]
  • Cono de tiro: La dirección Z se limita a X * tan(max_spread_deg). Con 5°, la bocha solo puede desviarse ~8.75% de su distancia recorrida lateralmente.
  • Potencia mínima: Si power < 0.05, el lanzamiento se ignora (evita tiros accidentales).
  • Umbral de curvatura: min_curve_threshold = 0.08 evita que gestos rectos o temblorosos generen efecto.
  • Velocidad mínima para spin: vel.length() > 0.1 — no se aplica spin si la bocha casi se detuvo.

Cambios Propuestos

Mejoras basadas en el gameplay: más control del jugador, menos resultados impredecibles, menor complejidad.

Cambio 1: Límite de dibujo — GestureTracker

Archivo: gesture_tracker.gd

Problema: No hay límite en el largo del dibujo. El jugador puede dibujar indefinidamente, generando perfiles de spin largos e impredecibles.

Solución: Agregar max_draw_points. Cuando se alcanza el límite, el gesto se completa automáticamente (se emite gesture_ended). Esto también limita naturalmente el tamaño del perfil de spin.

Nuevo @export

ParámetroDefaultDescripción
max_draw_points60Cantidad máxima de puntos del dibujo. Al alcanzarse, el lanzamiento se ejecuta automáticamente. 0 = sin límite.

Lógica

flowchart TD
  A[mouse motion: nuevo punto] --> B{max_draw_points > 0?}
  B -->|No| C[agregar punto normalmente]
  B -->|Sí| D{points.size >= max_draw_points?}
  D -->|No| C
  D -->|Sí| E["is_tracking = false\ngesture_ended.emit(points)"]

En _input, al agregar un punto por InputEventMouseMotion, verificar si se alcanzó el límite. Si sí, forzar el fin del gesto automáticamente — como si el jugador hubiera soltado el botón.

Cambio 2: Potencia bidireccional — GesturePower

Archivo: gesture_power.gd

Problema: peak_value solo sube. Si el jugador arrastra demasiado, no puede reducir la potencia. Se siente sin control.

Solución: Eliminar peak_value. Usar la distancia actual del drag directamente. La potencia sube al arrastrar hacia abajo y baja al arrastrar hacia arriba.

Variables eliminadas

peak_value: float = 0.0

Nueva lógica

flowchart TD
  A[gesture_started] --> B["bar.value = 0"]
  C[gesture_dragging] --> D["current = clamp(cur.y - start.y / max_drag_px, 0, 1) * 100"]
  D --> E["bar.value = current"]

Antes: bar.value = max(peak_value, current) — la barra solo subía, nunca bajaba.
Ahora: bar.value = current — la barra sigue el drag actual. Sube y baja libremente.

Gameplay: El jugador tiene control total de la potencia. Arrastra hacia abajo para cargar, suelta hacia arriba para reducir. Soltar en el momento justo es parte del skill.

Cambio 3: Spin simplificado — GestureSpin + ThrowHandler

Archivos: gesture_spin.gd, throw_handler.gd

Problema: El perfil de spin (array de valores) produce curvas impredecibles. La dirección y el efecto se mezclan de forma confusa. El sistema es complejo sin beneficio de gameplay.

Solución: Reemplazar el perfil completo por un único valor float. El spin se aplica como una rotación constante por frame, sin perfiles ni índices.

Gesto → Spin: GestureSpin simplificado

flowchart TD
  A[gesture_ended: points] --> B{points.size < 3?}
  B -->|Sí| C["emit spin_calculated(0.0)"]
  B -->|No| D["dir = (end - start).normalized()"]
  D --> E["perp = Vector2(-dir.y, dir.x)"]
  E --> F["Para cada punto: lateral = (p - start) · perp / sensitivity"]
  F --> G["curve = promedio(laterals)"]
  G --> H["curve = clamp(curve, -1.0, 1.0)"]
  H --> I["emit spin_calculated(curve)"]

En vez de emitir PackedFloat32Array, se emite float — un solo número que representa la curvatura promedio del gesto.

AntesDespués
spin_profile_calculated(values: PackedFloat32Array)spin_calculated(curve: float)
Array completo de desviacionesPromedio de desviaciones (-1 a 1)
Se消费e valor por valor a frecuencia spin_hzValor constante aplicado cada frame

Aplicación del spin: ThrowHandler simplificado

Variables eliminadas:

  • spin_profile: PackedFloat32Array
  • spin_idx: int
  • spin_timer: float

Variables nuevas:

  • spin_curve: float = 0.0 — valor de curvatura actual (-1 a 1)

Nuevos @export

ParámetroDefaultDescripción
steer_force1.5Fuerza de curvatura por frame. Controla cuánto efecto tiene la bocha. Reemplaza steer_strength.

@export eliminados

Parámetro eliminadoRazón
spin_hzNo se necesita frecuencia con un valor constante

Nueva lógica _on_spin

flowchart TD
  A[spin_calculated: curve] --> B{abs(curve) > min_curve_threshold?}
  B -->|No| C["spin_curve = 0.0\nis_applying_spin = false"]
  B -->|Sí| D["spin_curve = curve\nis_applying_spin = true"]

Nueva lógica _physics_process

flowchart TD
  A[_physics_process] --> B{is_applying_spin?}
  B -->|No| Z[return]
  B -->|Sí| C{vel.length > 0.1?}
  C -->|No| Z
  C -->|Sí| D["ball.linear_velocity =\n  vel.rotated(Vector3.UP, spin_curve * steer_force * delta)"]

Antes: Un perfil de N valores se consumía a frecuencia spin_hz, cada uno con su propia curvatura. Resultado impredecible.
Ahora: Un valor constante rota la velocidad suavemente cada frame. La curva es predecible: siempre la misma intensidad, siempre en la misma dirección. Lo que dibujaste es lo que obtenés.

Ideas Futuras

Anticipación visual

Mostrar la trayectoria predicha (parábola + curva) antes de soltar, actualizada en tiempo real durante el drag. Esto le daría al jugador feedback continuo.

Estado de lanzamiento explícito

El sistema no tiene un state machine explícito. Agregar estados (IDLE, AIMING, THROWING, ROLLING) facilitaría agregar lógica como reset, cámaras, y transiciones.

Flick bidireccional

El flick solo detecta "bajo → sube". Se podría soportar también "sube → baja" para lanzamientos con estilos diferentes, o gestos circulares para efectos especiales.
