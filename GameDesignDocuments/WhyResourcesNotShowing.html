Por que no aparecen los Resources en Godot

Explicacion del problema y 3 soluciones concretas

El Problema

Cuando haces click derecho en el FileSystem → New Resource... y buscas PositioningState, no aparece en la lista. Lo mismo con los otros 3 estados. Esto es un comportamiento conocido de Godot 4.x con class_name en clases que extienden RefCounted.

Por que pasa esto

class_name registra la clase globalmente para que puedas usarla en codigo (var x = PositioningState.new()), pero NO necesariamente la registra en el dialogo de "New Resource". Godot solo muestra ciertos tipos en ese dialogo, y las clases custom que extienden RefCounted a veces no aparecen, especialmente si:
  • El proyecto no recompilo los scripts despues de agregar el class_name
  • Hay un error de compilacion en algun script que bloquea el registro
  • Godot 4.x tiene un bug conocido con class_name + RefCounted en el resource picker

Solucion 1 (RECOMENDADA): Usar Opcion A - Crear estados en codigo

Esta es la solucion mas simple y la que te recomiendo. No necesitas crear archivos .tres ni asignarlos en el inspector. El StateMachine crea los estados automaticamente en su _ready().

Ventaja: Funciona inmediatamente. Zero configuracion en el editor.
Desventaja: Los @export en los states no se ven en ningun inspector (pero no rompen nada).

Paso unico: Reemplazar StateMachine.gd

# StateMachine.gd -- Version Opcion A (sin .tres, funciona YA)
extends Node

signal state_changed(new_state: String)

enum BochaState {
    POSITIONING,
    AIMING,
    THROWING,
    ROLLING
}

var current_state: BochaState = BochaState.POSITIONING

# Los estados se crean en codigo, NO desde el inspector
var states: Dictionary = {}

func _ready() -> void:
    # Crear los estados directamente con .new()
    states[BochaState.POSITIONING] = PositioningState.new()
    states[BochaState.AIMING] = AimingState.new()
    states[BochaState.THROWING] = ThrowingState.new()
    states[BochaState.ROLLING] = RollingState.new()

    # Inicializar cada estado con referencia al padre
    for state in states.values():
        state.init(get_parent())

    # Entrar al estado inicial
    states[current_state].enter()

func _process(delta: float) -> void:
    states[current_state].update(delta)

func _physics_process(delta: float) -> void:
    states[current_state].physics_update(delta)

func _input(event: InputEvent) -> void:
    states[current_state].handle_input(event)

func transition_to(new_state: BochaState) -> void:
    if new_state == current_state:
        return

    states[current_state].exit()
    current_state = new_state
    states[current_state].enter()
    state_changed.emit(BochaState.keys()[current_state])
Con esto ya funciona. No necesitas crear ningun .tres. Los @export que tienen los scripts de los estados simplemente se ignoran (no causan error, solo no se muestran en ningun inspector).

Solucion 2: Crear los .tres manualmente como archivos de texto

Si igual queres los archivos .tres para poder configurarlos desde el inspector despues, podés crearlos manualmente como archivos de texto. Godot los lee perfectamente.

Paso 1: Crear los 4 archivos .tres en la carpeta States/

Creá estos archivos de texto en res://BochaGian/States/:

positioning_state.tres

[gd_resource type="PositioningState" format=3]

freeze_on_enter = true
reset_velocity_on_enter = true

aiming_state.tres

[gd_resource type="AimingState" format=3]

show_power_indicator = true
min_power_to_throw = 0.1

throwing_state.tres

[gd_resource type="ThrowingState" format=3]

apply_curve = true
curve_multiplier = 2.0
play_throw_effect = true

rolling_state.tres

[gd_resource type="RollingState" format=3]

stop_threshold = 0.05
required_stop_time = 0.5
enable_trail_effect = true
trail_min_speed = 0.5

Paso 2: Asignar los .tres al StateMachine desde el inspector

  1. Abrí bocha.tscn
  2. Seleccioná el nodo StateMachine
  3. En el Inspector, seccion States
  4. Para cada campo (Positioning State, Aiming State, etc.):
    • Hace click en el campo vacio
    • Elegí Load...
    • Navegá a res://BochaGian/States/
    • Seleccioná el .tres correspondiente
  5. Guardá la escena (Ctrl+S)
Importante: Despues de crear los archivos .tres, Godot necesita importarlos. Si no aparecen en el FileSystem, hace click en el boton Scan (icono de carpeta con flechas) en la parte superior del panel FileSystem, o reinicia el proyecto.

Paso 3: Mantener el StateMachine.gd de la Opcion B

Si usas esta solucion, el StateMachine.gd se queda como ya lo tenes (con los @export var). No hace falta cambiarlo.

Solucion 3: Forzar que aparezcan en el dialogo de New Resource

Si queres que los states aparezcan en el dialogo de New Resource..., hay un par de cosas que podes probar:

Intento A: Recargar los scripts del proyecto

  1. En Godot, anda a Project → Reload Current Project
  2. Espera a que recargue
  3. Intenta de nuevo: click derecho en FileSystem → New Resource → busca "PositioningState"

Intento B: Verificar que no haya errores de compilacion

Si algun script tiene un error, Godot no registra los class_name. Fijate en el Output del editor si hay errores de compilacion. Los scripts actuales de BochaGian estan bien, pero si agregaste algo nuevo, puede estar bloqueando el registro.

Intento C: Usar @icon para que aparezcan mejor

Agregar un icono al class_name a veces ayuda a que Godot lo reconozca mejor en los dialogos: 
extends RefCounted
class_name PositioningState "res://icon.svg"
# ^ El icono es opcional, pero a veces ayuda

Comparacion de las 3 soluciones

Mi recomendacion

Usa la Solucion 1 (Opcion A). Cambia el StateMachine.gd al que crea los estados con .new() y listo. Funciona inmediatamente, sin configuracion extra.

Despues, cuando todo el juego este funcionando y quieras ajustar valores de los estados frecuentemente, recien ahi migras a la Opcion B creando los .tres (Solucion 2). No tiene sentido pelear con el editor de Godot para algo que no es esencial en esta etapa.

El patron de State Machine funciona igual de bien con ambas opciones. La unica diferencia es donde se configuran los valores (codigo vs inspector).
